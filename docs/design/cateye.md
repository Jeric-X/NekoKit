# Cateye 图片识别工具集设计文档

## 设计目标

为 AI 智能体提供一站式图片理解能力。图片理解是一个多维度的任务——文字提取、来源追溯、语义理解各有不同的技术路线和最佳实践。Cateye 将这些能力拆解为独立工具，让 AI 能够根据用户意图精准选择，避免"一把大锤敲所有钉子"的低效模式。

---

## 核心理念

### 工具解耦

将图片理解拆分为 5 个独立工具，各司其职：

| 工具 | 类型 | 职责边界 |
|------|------|----------|
| `cateye_ocr` | 核心 | 文字提取，只负责把图中的文字读出来 |
| `cateye_search` | 核心 | 以图搜图，只负责找到图片的来源或相似图 |
| `cateye_vision` | 核心 | 视觉理解，只负责用大模型解读图片内容 |
| `cateye_preprocess` | 辅助 | 图片预处理，按任务类型优化尺寸和格式 |
| `cateye_cache` | 辅助 | 缓存管理，避免对相同图片重复调用 API |

解耦的好处是 AI 可以按需组合：用户只想提取文字时，不必触发大模型调用；只想找图片来源时，不必等待 OCR 处理。每个工具的参数定义简洁明确，降低 AI 的决策成本。

### 共享缓存

5 个工具共享同一个 `ImageCache` 实例，通过 MD5 精确匹配 + dHash 相似图检测实现智能缓存：

- **MD5 匹配**：对图片二进制数据计算 MD5，完全相同的图片直接命中缓存
- **dHash 相似图检测**：对图片计算感知哈希（dHash），通过汉明距离判断相似度。同一张图的不同压缩版本、轻微裁切版本都能命中缓存

缓存按任务类型（ocr/search/vision）分区存储，同一张图片的 OCR 结果不会与搜图结果混淆。

### 自动预处理

核心工具（OCR、搜图、视觉理解）在执行时自动检测配置，若预处理开关开启，会自动调用预处理流程优化图片。不同任务类型有不同的预设参数：

| 任务类型 | 最大尺寸 | 输出格式 | 特殊处理 |
|----------|----------|----------|----------|
| OCR | 2048px | PNG | 灰度化 |
| 搜图 | 1024px | JPEG (质量85) | — |
| 视觉理解 | 2048px | JPEG (质量90) | — |

预处理对 AI 透明——AI 无需手动调用 `cateye_preprocess`，核心工具会自动完成。`cateye_preprocess` 作为独立工具暴露，供 AI 在需要单独调试预处理效果时使用。

### 供应商抽象

搜图功能通过供应商抽象层实现可插拔。当前支持三个供应商：

| 供应商 | 适用场景 | 接入方式 |
|--------|----------|----------|
| trace.moe | 番剧截图识别 | 免费 API，上传图片搜索 |
| SauceNAO | 插画/漫画/二次元 | API Key 可选，免费版限额较少 |
| 华为云 | 通用图片搜索 | 需要 API Key + 项目 ID |

AI 通过 `scene` 参数指定场景类型（anime/moe/illustration/general），工具自动映射到最合适的供应商。也可以通过 `provider` 参数强制指定供应商。`auto` 模式会依次尝试所有已启用的供应商，合并结果。

## 三核心工具的设计思路

### OCR（cateye_ocr）

**核心挑战**：EasyOCR 是同步阻塞库，直接在异步事件循环中调用会阻塞整个服务。

**设计决策**：
- 使用 `run_in_executor` 将 EasyOCR 的 `readtext` 调用委托给线程池，避免阻塞异步事件循环
- Reader 实例懒加载并缓存，首次调用时初始化，后续调用复用，避免重复加载模型
- 语言列表支持通过参数覆盖配置默认值，AI 可以根据图片内容动态选择识别语言

**缓存策略**：OCR 结果按图片 MD5 + dHash 缓存，同一张图或相似图不重复识别。

### 搜图（cateye_search）

**核心挑战**：不同供应商的 API 协议、认证方式、返回格式各不相同。

**设计决策**：
- 供应商信息集中定义在 `PROVIDERS` 字典中，包含名称、URL、适用场景
- 场景到供应商的映射由 `SCENE_PROVIDER_MAP` 管理，新增场景只需添加映射条目
- 每个供应商的 API 调用逻辑封装在独立的 `_call_xxx` 方法中，互不干扰
- `_resolve_providers` 方法统一处理 scene 和 provider 参数的优先级，provider 参数优先级高于 scene
- 所有供应商调用失败时，返回空结果而非报错，让 AI 可以尝试其他工具（如 vision）作为降级方案

### 视觉理解（cateye_vision）

**核心挑战**：不同复杂度的图片理解任务需要不同级别的模型，但 AI 不应关心模型选择细节。

**设计决策**：
- 双模式设计：`daily` 模式用于日常场景（表情包、日常照片），`professional` 模式用于复杂分析（图表、解题）
- 模型选择由配置驱动，管理员在 WebUI 中为两种模式分别配置 Provider ID
- `professional_model` 留空时自动降级为 `daily_model`，确保功能可用
- 通过 AstrBot 的 `llm_generate` 接口调用平台 LLM，传入图片的 Base64 编码和系统提示词
- 系统提示词根据模式动态构建：daily 模式简洁友好，professional 模式强调精确和详尽

## 两辅助工具的设计思路

### 预处理（cateye_preprocess）

**设计定位**：作为独立工具暴露，但主要供核心工具内部自动调用。

**预设参数表**：在 `_internal.py` 中以字典形式定义三种任务的预处理参数。参数选择基于实践经验——OCR 需要灰度化提升文字对比度，搜图需要压缩到较小尺寸加快上传速度，视觉理解需要在质量和 token 消耗间取平衡。

**独立暴露的意义**：AI 可以在不确定图片是否需要预处理时，先用 `cateye_preprocess` 检查效果，再决定是否调用核心工具。也方便调试预处理参数。

### 缓存（cateye_cache）

**设计定位**：作为独立工具暴露，但主要供核心工具内部自动调用。

**TTL 机制**：缓存条目带有时间戳，读取时检查是否过期。过期条目在读取时惰性删除，`cleanup` 方法可主动清理所有过期条目。

**汉明距离阈值**：默认阈值为 5，两张图片的 dHash 汉明距离不超过 5 即视为相似图。这个阈值在"误命中率"和"漏检率"之间取得了较好的平衡——同一张图的不同压缩版本通常汉明距离在 1-3 之间，而不同内容的图片通常在 10 以上。

**独立暴露的意义**：AI 可以在调用核心工具前先用 `cateye_cache(action="check")` 检查缓存，命中时直接使用缓存结果，避免不必要的 API 调用。也可以在获得结果后主动 `store`，为后续相同图片的查询做准备。

## 工具组合策略

Cateye 的工具组合遵循"按意图选择，按需组合"的原则：

| 用户意图 | 推荐组合 | 理由 |
|----------|----------|------|
| 提取文字 | OCR 单独使用 | 文字提取不需要其他能力 |
| 识别角色/人物 | 搜图 + 视觉理解 | 搜图找来源，视觉理解描述外观 |
| 查找番剧出处 | 搜图（anime 场景） | trace.moe 专精番剧识别 |
| 查找插画来源 | 搜图（moe 场景） | SauceNAO 专精二次元插画 |
| 理解表情包 | 视觉理解（daily 模式） | 日常模式足够且省 token |
| 分析图表/解题 | 视觉理解（professional 模式） | 专业模式提供精确分析 |
| 通用"这是什么" | OCR + 搜图 + 视觉理解 | 全面分析，不遗漏任何维度 |

核心工具内部已集成自动预处理和缓存检查，AI 在大多数场景下只需调用核心工具，无需手动编排辅助工具。

## 扩展性设计

### 新增搜图供应商

在 `PROVIDERS` 字典中添加供应商信息，在 `SCENE_PROVIDER_MAP` 中添加场景映射，实现 `_call_xxx` 方法，在 `_call_provider` 中添加分发分支。供应商的启用/禁用通过配置项控制，无需修改代码。

### 新增视觉理解模式

在 `VisionTool._build_system_prompt` 中添加新的模式分支，在参数定义的 `enum` 中添加新模式名称，在配置 Schema 中添加对应的模型配置项。例如可以添加"creative"模式用于创意描述。

### 新增图片理解能力

在 `tools/image_analyzer/` 下创建新的工具模块，继承 `BaseTool` 实现业务逻辑，在 `__init__.py` 中导出，在 `main.py` 中创建对应的 FunctionTool 子类并注册。例如可以添加"图片风格迁移"或"图片超分辨率"工具。

### 替换 OCR 引擎

当前 OCR 基于 EasyOCR，如需替换为 PaddleOCR 或云服务 OCR，只需修改 `OCRTool._get_reader` 和 `execute` 中的识别逻辑，工具接口（参数、返回格式）保持不变，AI 侧无需任何调整。

### 缓存后端升级

当前 `ImageCache` 使用内存字典存储，重启后缓存丢失。如需持久化缓存，可以将 `ImageCache` 改为基于 SQLite 或 Redis 的实现，只需保持相同的 `get/store/find_similar` 接口即可。
