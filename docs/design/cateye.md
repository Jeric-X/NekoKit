# Cateye 图片识别工具集设计文档

## 设计目标

为 AI 智能体提供一站式图片理解能力。图片理解是一个多维度的任务——文字提取、来源追溯、语义理解各有不同的技术路线和最佳实践。Cateye 将这些能力拆解为独立工具，让 AI 能够根据用户意图精准选择，避免"一把大锤敲所有钉子"的低效模式。

---

## 核心理念

### 工具解耦

将图片理解拆分为 6 个独立工具，各司其职：

| 工具 | 类型 | 职责边界 |
|------|------|----------|
| `cateye_ocr` | 核心 | 文字提取，只负责把图中的文字读出来 |
| `cateye_search` | 核心 | 以图搜图，只负责找到图片的来源或相似图 |
| `cateye_vision` | 核心 | 视觉理解，只负责用大模型解读图片内容 |
| `cateye_preprocess` | 辅助 | 图片预处理，按任务类型优化尺寸和格式 |
| `cateye_cache` | 辅助 | 缓存管理，避免对相同图片重复调用 API |
| `cateye_scene` | 编排 | 场景预设，返回工具组合策略 |

解耦的好处是 AI 可以按需组合：用户只想提取文字时，不必触发大模型调用；只想找图片来源时，不必等待 OCR 处理。每个工具的参数定义简洁明确，降低 AI 的决策成本。

### 核心与辅助分离

核心工具（OCR、搜图、视觉理解）专注于识别逻辑，**不内置缓存和预处理**。这种设计确保：

- **职责单一**：核心工具只做识别，不承担缓存和优化职责
- **灵活组合**：智能体可以根据场景自由决定是否使用缓存和预处理
- **可调试性**：缓存命中/未命中、预处理效果对智能体完全透明

辅助工具由智能体根据场景预设方案按步骤调用，而非核心工具内部自动触发。

### 场景预设

`cateye_scene` 提供场景预设功能，将工具组合策略以 JSON 格式存储。每个预设包含：

- `name`：场景名称
- `description`：场景描述
- `steps`：工具调用步骤列表，每步指定 `tool`（工具名）和 `params`（固定参数）

预设存储在 `{data_dir}/cateye/presets/scene_presets.json`，智能体可通过 `update` 操作自行修改方案，实现任务自适应。

### 共享缓存

辅助工具 `cateye_cache` 使用 `ImageCache` 实例，通过 MD5 精确匹配 + dHash 相似图检测实现智能缓存：

- **MD5 匹配**：对图片二进制数据计算 MD5，完全相同的图片直接命中缓存
- **dHash 相似图检测**：对图片计算感知哈希（dHash），通过比特级汉明距离判断相似度。同一张图的不同压缩版本、轻微裁切版本都能命中缓存

缓存按任务类型（ocr/search/vision）分区存储，同一张图片的 OCR 结果不会与搜图结果混淆。

### 供应商抽象

搜图功能通过供应商抽象层实现可插拔。当前支持三个供应商：

| 供应商 | 适用场景 | 接入方式 |
|--------|----------|----------|
| trace.moe | 番剧截图识别 | 免费 API，上传图片搜索 |
| SauceNAO | 插画/漫画/二次元 | API Key 可选，免费版限额较少 |
| 华为云 | 通用图片搜索 | 需要 API Key + 项目 ID |

AI 通过 `scene` 参数指定场景类型（anime/moe/illustration/general），工具自动映射到最合适的供应商。也可以通过 `provider` 参数强制指定供应商。`auto` 模式会依次尝试所有已启用的供应商，合并结果。

## 三核心工具的设计思路

### OCR（cateye_ocr）

**引擎选择**：使用 RapidOCR（基于 ONNX Runtime），相比 EasyOCR 的优势：
- 无需 GPU 即可高效运行
- 默认模型已支持中英文，无需指定语言列表
- 初始化更快，内存占用更低

**设计决策**：
- 使用 `run_in_executor` 将 RapidOCR 的同步调用委托给线程池，避免阻塞异步事件循环
- 引擎实例懒加载并缓存，首次调用时初始化，后续调用复用
- 通过 `text_score`（文本置信度阈值）配置项控制识别精度，低于阈值的文本块被过滤

### 搜图（cateye_search）

**核心挑战**：不同供应商的 API 协议、认证方式、返回格式各不相同。

**设计决策**：
- 供应商信息集中定义在 `PROVIDERS` 字典中，包含名称、URL、适用场景
- 场景到供应商的映射由 `SCENE_PROVIDER_MAP` 管理，新增场景只需添加映射条目
- 每个供应商的 API 调用逻辑封装在独立的 `_call_xxx` 方法中，互不干扰
- `_resolve_providers` 方法统一处理 scene 和 provider 参数的优先级，provider 参数优先级高于 scene
- 所有供应商调用失败时，返回空结果而非报错，让 AI 可以尝试其他工具（如 vision）作为降级方案

### 视觉理解（cateye_vision）

**核心挑战**：不同复杂度的图片理解任务需要不同级别的模型，但 AI 不应关心模型选择细节。

**设计决策**：
- 双模式设计：`daily` 模式用于日常场景（表情包、日常照片），`professional` 模式用于复杂分析（图表、解题）
- 模型选择由配置驱动，管理员在 WebUI 中为两种模式分别配置 Provider ID
- `professional_model` 留空时自动降级为 `daily_model`，确保功能可用
- 通过 AstrBot 的 `llm_generate` 接口调用平台 LLM，传入图片的 Base64 编码和系统提示词
- 系统提示词根据模式动态构建：daily 模式简洁友好，professional 模式强调精确和详尽

## 场景预设的设计思路

### 设计动机

核心工具与辅助工具解耦后，智能体需要知道如何正确组合工具。场景预设将常见的组合策略固化为可复用的方案，降低智能体的编排负担。

### JSON 格式

```json
{
  "name": "场景名称",
  "description": "场景描述",
  "steps": [
    {"tool": "cateye_cache", "params": {"task_type": "ocr", "action": "check"}},
    {"tool": "cateye_preprocess", "params": {"task_type": "ocr"}},
    {"tool": "cateye_ocr", "params": {}},
    {"tool": "cateye_cache", "params": {"task_type": "ocr", "action": "store"}}
  ]
}
```

每个 step 的 `params` 中不包含 `image_url`，该参数由运行时自动注入。

### 自适应机制

智能体可通过 `cateye_scene(action="update")` 修改预设方案。例如：
- 发现某个场景的预处理步骤不必要时，可以删除对应 step
- 需要为特定任务组合多个核心工具时，可以新增 step
- 可以创建全新的场景编码，扩展预设库

## 两辅助工具的设计思路

### 预处理（cateye_preprocess）

**设计定位**：独立工具，由智能体根据场景预设方案按需调用。

**预设参数表**：在 `_internal.py` 中以字典形式定义三种任务的预处理参数。参数选择基于实践经验——OCR 需要灰度化提升文字对比度，搜图需要压缩到较小尺寸加快上传速度，视觉理解需要在质量和 token 消耗间取平衡。

### 缓存（cateye_cache）

**设计定位**：独立工具，由智能体根据场景预设方案按需调用。

**TTL 机制**：缓存条目带有时间戳，读取时检查是否过期。过期条目在读取时惰性删除，`cleanup` 方法可主动清理所有过期条目。

**汉明距离计算**：dHash 产生十六进制字符串，汉明距离通过 XOR + 统计比特位 `1` 的个数计算（而非字符级比较），确保距离度量的准确性。空 dHash 时跳过相似度匹配，仅依赖 MD5 精确匹配。

## 扩展性设计

### 新增搜图供应商

在 `PROVIDERS` 字典中添加供应商信息，在 `SCENE_PROVIDER_MAP` 中添加场景映射，实现 `_call_xxx` 方法，在 `_call_provider` 中添加分发分支。供应商的启用/禁用通过配置项控制，无需修改代码。

### 新增视觉理解模式

在 `VisionTool._build_system_prompt` 中添加新的模式分支，在参数定义的 `enum` 中添加新模式名称，在配置 Schema 中添加对应的模型配置项。

### 新增图片理解能力

在 `tools/image_analyzer/` 下创建新的工具模块，继承 `BaseTool` 实现业务逻辑，在 `__init__.py` 中导出，在 `main.py` 中创建对应的 FunctionTool 子类并注册。

### 替换 OCR 引擎

当前 OCR 基于 RapidOCR，如需替换为其他引擎，只需修改 `OCRTool._get_engine` 和 `execute` 中的识别逻辑，工具接口（参数、返回格式）保持不变，AI 侧无需任何调整。

### 新增场景预设

通过 `cateye_scene(action="update")` 动态添加，或修改 `scene_preset_tool.py` 中的 `DEFAULT_PRESETS` 字典添加内置预设。
